# EasyJet — Продуктовая документация

## Обзор

**EasyJet** — это легковесная система непрерывной интеграции и доставки (CI/CD), написанная на Go. Проект позиционируется как аналог Jenkins с минимизированным функционалом, ориентированный на пет-проекты и небольшие команды.

## Ключевые особенности

### Производительность и ресурсы

- **Минимальное потребление ресурсов** — работает на любом оборудовании, включая Raspberry Pi
- **Одноузловая архитектура** — все компоненты работают в рамках одного сервера

### Функциональность

- **Простота использования** — малый и понятный набор функций
- **Быстрый деплой** — возможность быстро поднять типовой репозиторий
- **Гибкость выполнения** — поддержка bash-скриптов и Docker-контейнеров
- **Работа с секретами** — проброс переменных окружения для скриптов

### Интеграции

- **Git-интеграция** — получение изменений и git log
- **Gitea integration** — статусы коммитов
- **Prometheus** — метрики потребления ресурсов и состояния приложения
- **Telegram** — алерты и уведомления
- **SSO через OpenID** — единая система аутентификации

## Этапы развития

### Этап 1: MVP (Минимально жизнеспособный продукт)

**Цель:** Создать базовую работающую систему для выполнения задач по расписанию и вручную.

#### Функционал

- **Сборка Docker-образов** — через shell-скрипты
- **Управление Docker Compose** — поднятие compose-файлов через shell
- **Выполнение команд** — запуск через `/bin/sh -xe`
- **Планировщик задач** — запуск джоб:
  - По cron-расписанию
  - Вручную
- **Управление проектами:**
  - Название проекта
  - Настройка Git-репозитория (ссылка + ветка)
  - Рабочая директория (автоматически)
  - Настройка cron
  - Несколько этапов (скриптов) для задачи
  - История запусков с ротацией
- **Git-интеграция** — вывод списка изменений
- **Режим демона** — запуск как Linux-демон (под пользователем)
- **База данных** — поддержка SQLite и PostgreSQL

#### Технические требования

- Работа в режиме единственного узла
- Docker-среда для выполнения задач
- Веб-интерфейс для настройки и управления

### Этап 2: Расширение функционала

**Цель:** Добавить мониторинг, алертинг и улучшенное управление деплоем.

#### Функционал

- **Мониторинг и метрики:**
  - Интеграция с Grafana
  - Метрики Prometheus:
    - Потребление CPU и RAM
    - Метрики GC
    - Метрики самого приложения
- **Алертинг:**
  - Уведомления в Telegram
- **Управление конфигурацией:**
  - Создание Docker Compose и прочих файлов из сервиса
  - Возможность шифрования файлов
  - Поддержка перезаписи файлов
- **Типовой деплой**:
  - Выбор директории для размещения compose-файлов
  - Создание необходимых файлов
  - Проверка существующих файлов с предупреждением о несовпадении хешей
- **Самообновление:**
  - Второй демон для автообновления
  - Перезапуск основного пода через API

### Этап 3: Шаблоны и совместная работа

**Цель:** Обеспечить возможность шаринга конфигураций деплоев.

#### Функционал

- **Система шаблонов:**
  - С поддержкой переменных
- **Расширенная интеграция с Gitea:**
  - Статусы сборок в интерфейсе Gitea
- **Улучшенная работа с Docker:**
  - Поддержка Docker-in-Docker (DinD)
  - Агентная система для изоляции задач

### Этап 4: Дальнейшее развитие

Потенциальный функционал:

- **Веб-хуки** — триггеры сборок от Git-сервисов
- **Распределённая архитектура** — поддержка нескольких узлов
- **Ролевая модель** — разграничение прав доступа
- **Аудит и логирование** — детальное логирование действий
- **API для интеграций** — публичный REST API

## Цели проекта

### Краткосрочные

1. Создать работающий MVP с базовым функционалом CI/CD
2. Обеспечить минимальное потребление ресурсов
3. Предоставить простой и понятный веб-интерфейс

### Долгосрочные

1. Стать легковесной альтернативой Jenkins для пет-проектов
2. Создать экосистему шаблонов для типовых деплоев
3. Обеспечить бесшовную интеграцию с Gitea и другими Git-сервисами

## Архитектура данных

### Основные сущности

| Сущность                 | Назначение                                  |
| ------------------------ | ------------------------------------------- |
| Проект                   | Конфигурация пайплайна для репозитория      |
| Этап проекта             | Один этап пайплайна (скрипт для выполнения) |
| Запуск проекта           | Результат выполнения пайплайна (запуск)     |
| Результат этапа запуска  | Результат выполнения конкретного этапа      |
| Git-логи запуска проекта | Git-коммиты, связанные с запуском           |

### Ключевые бизнес-процессы

1. **Регистрация нового проекта** — создание конфигурации пайплайна
2. **Запуск развёртывания** — ручная активация выполнения пайплайна
3. **Обработка ожидающих развёртываний** — фоновое выполнение задач
4. **Получение информации о проекте** — просмотр конфигурации
5. **Получение истории запусков** — просмотр результатов выполнения
6. **Обновление конфигурации проекта** — редактирование настроек
