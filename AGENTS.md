## Обязательные действия после модификации кода

После **любого** изменения Go кода:

2. Выполни **все** команды через `task` как указано в `docs/go/rules.md`
3. Убедись что все команды прошли успешно

## Обязательные требования к бизнес-документации

### При создании или изменении DFD-диаграмм

1. Прочитай `docs/business/rules.md` — раздел «DFD-диаграммы»
2. Соблюдай правила нотации Mermaid:
   - Внешние сущности: `User[Название]`
   - Процессы: `Process1([1.0 Название])`
   - Хранилища: `DB1[(D1: Название)]`
   - Потоки: `User -->|Что происходит| Process1`
3. **Обязательно объявляй все элементы перед описанием потоков**
4. В потоках данных используй только имена элементов (без повторения нотации)
5. Пиши описания на русском языке бизнес-терминами
6. Убедись что диаграмма следует шаблону из `docs/business/rules.md`

### При описании бизнес-сущностей (`entity.md`)

1. Прочитай `docs/business/rules.md` — раздел «Описание бизнес-сущностей»
2. Для каждой сущности указывай:
   - Бизнес-название с CodeName в заголовке
   - Название в коде (жирным в бэкитиках)
   - Бизнес-описание (2-4 предложения)
   - Таблицу полей с бизнес-назначением
3. Добавляй ER-диаграмму связей между сущностями
4. Описывай связи на русском языке (например, «имеет этапы», «имеет запуски»)

### При описании экранов приложения (`screens.md`)

1. Прочитай `docs/business/rules.md` — раздел «Описание экранов приложения»
2. Для каждого экрана указывай:
   - Порядковый номер и название
   - Путь (URL) и компонент
   - Бизнес-описание назначения экрана
   - Используемые сущности
   - Действия пользователя с результатами
   - API-эндпоинты
3. Добавляй диаграмму навигации между экранами
4. Веди сводную таблицу всех экранов в конце файла

### Запрещено при работе с бизнес-документацией

- ❌ Использовать технические термины в названиях процессов и потоков
- ❌ Повторять нотацию в описании потоков данных
- ❌ Пропускать объявление элементов перед описанием потоков
- ❌ Описывать поля сущностей без бизнес-назначения
- ❌ Пропускать ER-диаграммы связей между сущностями
- ❌ Игнорировать правила из `docs/business/rules.md`

## Запрещено

- ❌ Изменять код без последующего выполнения `task go:format`, `task go:lint`, `task go:test`, `task build:server`
- ❌ Изменять код без последующего выполнения `task ts:format`, `task ts:lint`, `task build:front`
- ❌ Изменять документацию без последующего выполнения `task md:format`, `task md:lint`
- ❌ Запускать команды напрямую вместо `task`
- ❌ Игнорировать ошибки линтера или сборки
